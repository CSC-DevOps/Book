Introduction

About this book

Software developers no longer simply build software in isolation: They now are expected to continuously deploy fixes and experimental features to production environments serving millions of customers. Making such ultra-fast and automatic changes to production means that testing and verifying the design and implementation of computing environments is increasingly important. However, testing can be lacking[report], and environment construction is easy to get wrong. For example, GitLab lost 300 GB of customer data after accidentally deleting their production database[gitlab]. Even worse, they could not restore the data because they discovered their backup procedure had been failing due to a mismatch in versions between the dump utility (pg_dump 9.2) and their database (PostgreSQL 9.6).

Unfortunately, the skills required to construct and test these computing environments requires expertise and training that is even more rare and highly sought than data science skills[survey]. For example, Mozilla’s Kim Moir says she “recently looked at the undergrad classes required to graduate with a computer science degree from a major university, and [she] was struck by [a lack of] practice on deploying code. In most computer science programs, there is little emphasis on infrastructure”. Similarly, Google’s Boris Debic claims that “Release engineering is not taught; it’s often not even mentioned in courses where it should be mentioned”. For this reason, Facebook’s Chuck Rossi considers hiring engineers with these skills “is like finding unicorns.”[Adams:2015]

This book was born out of experiences teaching over 1000 university students, consulting with dozens of software development companies at the fore-front of continuous deployment, performing empirical research on software engineering, and conversations with software giants such as Kent Beck, Jez Humble, and Leigh Honeywell. You will have trouble finding the latest devops related tools, such as helm, kubernetes, and kafta in the book—hopefully, because they have been replaced with something better by the time you are reading this. But what should remain are essential concepts needed to understand, apply, and even build these tools yourselves.

Consider one homework assignment, where students used configuration management tools to install and configure an open-source chat server called Mattermost, which works much like Slack. The computing environment for Mattermost requires several components: a database, system dependencies, the Mattermost server itself, and several configuration files systemd services, mysql.cnf, and config.json for Mattermost). In configuring this environment, many things could go wrong. For example, a simple typo or malformed JSON in a configuration file could result in an non-functioning environment, but with little hints as to why. To diagnose this problem, students might need to check a variety of system components using a myriad of tools and shell utilities in which they have little experience, such as the mysql shell, systemctl, journalctl, cat, grep, and jq. In response, we would have to ask a series of questions: “did you check your mysql credentials,” “did you check your connection string is correct,” “did you run jsonlint on your configuration file.” Other times, strange behaviors would result from incidental factors, which we would only resolve after asking, “did you check dns,” “did you check your VM’s memory size.”

Diving head first into this world, there is too much to learn, too much that can go wrong, and very little concepts to build on. Let’s fix that. Read on, and learn with me.
